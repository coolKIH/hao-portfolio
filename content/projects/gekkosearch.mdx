## Overview

GekkoSearch is a financial data SaaS platform mainly focused on the Hong Kong market. It aggregates and structures data from sources such as HKEX and other financial datasets, connecting documents with related entities such as companies, people, and events.

The platform is designed to help professionals quickly locate relevant information and derive insights from large volumes of financial data.

## Case Study: Snippet → PDF Highlight

### Problem

Financial PDFs are difficult to work with:

- Content is unstructured and hard to search  
- Layouts are complex (tables, multi-column, mixed languages)  
- Search results are not naturally linked to document positions  
- Rendering (PDF viewer) is disconnected from search systems  

We needed to let users **search → preview → locate → interact** within documents.

### Solution

Designed the pipeline to connect search, text extraction, and document rendering:

```
PDF → Text → Search → Snippet → Position Mapping → PDF Highlight
```

This enables direct navigation from search results to the corresponding location in the document.

#### Making PDFs Searchable

- Extracted text using PDF.js in Node.js
- **Built a custom converter** for batch processing and structured outputs

Challenge: text is often fragmented  
Solution: reconstructed words based on spatial relationships

#### Snippet Navigation

Designed and implemented a position mapping system for snippet navigation.

- Injected position markers (e.g. page number, line number, detected table) into text  
- Filtered out markers during Elasticsearch indexing via a custom `char_filter` so they don’t affect search, while keeping them for frontend snippet rendering
- Implemented frontend rendering to display snippet location and support direct navigation to the corresponding document position with highlighting

#### Mapping Snippets Back to PDF

The core challenge was matching snippet text to rendered HTML.

- Used PDF.js text layer (HTML-based)
- Generated regex tolerant of HTML structure (spans, styles, layout nodes)
- Wrapped matched content with custom `<span>` for highlighting

#### Handling Edge Cases

- **Multi-page snippets:** matched per page with context-aware selection  
- **Lazy rendering:** re-applied highlights on PDF.js render events  
- **Content mismatch:** used tolerant matching to ignore layout noise  

#### Comments & Annotations

- Supported user text selection, highlighting, and commenting  
- Reused the same mapping logic for persistence and navigation  

### Trade-offs & Limitations

- Relies heavily on DOM manipulation  
- Tight coupling with PDF.js  
- Viewer logic became large and difficult to maintain  

### What I Would Improve

- Modularize viewer architecture  
- Extend or fork PDF.js instead of patching via DOM  
- Introduce structured position indexing  
- Improve separation of concerns  

### Impact

- Improved navigation and usability of financial documents  
- Enabled workflows such as highlighting and commenting  
- Provided a foundation for further document-based features
